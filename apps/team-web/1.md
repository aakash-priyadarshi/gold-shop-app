## MODULE PURPOSE

This module is the real-time audio backbone of the AI Sales Agent.
It manages the full duplex audio stream between:

Twilio WebSocket (raw audio in/out)
↕
Deepgram (speech-to-text, streaming)
↕
Claude LLM (conversation brain, streaming)
↕
ElevenLabs (text-to-speech, streaming)

The #1 engineering goal is PERCEIVED LATENCY UNDER 400ms.
The customer must never hear silence longer than 300-400ms.
Filler audio bridges that gap — it plays in 0ms while LLM thinks.

---

## PERFORMANCE TARGETS — NEVER REGRESS BELOW THESE

| Metric                          | Target  | Hard Limit |
| ------------------------------- | ------- | ---------- |
| End-of-utterance detection      | < 50ms  | < 80ms     |
| Filler audio playback start     | < 30ms  | < 50ms     |
| LLM first token (streaming)     | < 500ms | < 700ms    |
| TTS first audio chunk           | < 250ms | < 350ms    |
| Total perceived latency         | < 400ms | < 600ms    |
| Language switch detection       | < 100ms | < 150ms    |
| Backchannel audio injection     | < 20ms  | < 40ms     |
| Concurrent calls (per instance) | 50      | 100        |

## ARCHITECTURE OVERVIEW

### The Two Parallel Tracks (CRITICAL PATTERN)

When end-of-utterance is detected, ALWAYS fire two async operations
simultaneously — never sequentially:

```typescript
// ✅ CORRECT — parallel execution
const [_, llmStream] = await Promise.all([
  fillerAudioManager.playImmediate(context), // 0ms — plays instantly
  llmClient.streamResponse(transcript, context), // ~500ms — thinking
]);

// ❌ WRONG — sequential kills the latency target
await fillerAudioManager.playImmediate(context);
const llmStream = await llmClient.streamResponse(transcript, context);
```

### Full Call Flow

```
1. Twilio opens WebSocket → CallSession created
2. Raw mulaw audio chunks arrive → DeepgramSTT.feed(chunk)
3. Deepgram emits partial transcripts → SentencePredictor.update()
4. SentencePredictor detects near-complete thought → pre-warm LLM
5. Deepgram fires utterance_end → AudioPipeline.onUtteranceEnd()
6. PARALLEL:
   a. FillerAudioManager.playImmediate(emotional_context) → Twilio
   b. ConversationBrain.streamResponse(transcript) → LLM API
7. LLM first token arrives → ElevenLabsTTS.streamChunked(text)
8. TTS first audio chunk arrives → CrossfadeManager.transition()
9. Filler audio fades out, TTS response fades in seamlessly
10. Full response plays → pipeline returns to listening state
```

## CLASS DEFINITIONS

### AudioPipeline (main orchestrator)

```typescript
/**
 * Main orchestrator for the full duplex audio pipeline.
 * One instance per active call session.
 *
 * Lifecycle: created on call start, destroyed on call end.
 * Must be resilient — any component failure should degrade gracefully,
 * never crash the call.
 */
class AudioPipeline {
  private callId: string;
  private leadId: string;
  private twilioWs: WebSocket;
  private deepgramClient: DeepgramStreamingClient;
  private llmClient: ClaudeStreamingClient;
  private ttsClient: ElevenLabsStreamingClient;
  private fillerManager: FillerAudioManager;
  private crossfadeManager: CrossfadeManager;
  private backchannelEngine: BackchannelEngine;
  private sentencePredictor: SentencePredictor;
  private state: PipelineState;

  /**
   * Entry point for all incoming audio from Twilio.
   * Called on every WebSocket message — must be non-blocking.
   * Never await anything heavy here — use event emitters.
   */
  async onIncomingAudio(chunk: Buffer): Promise<void>;

  /**
   * Fired by Deepgram when utterance is complete.
   * This is the trigger point for the parallel filler + LLM pattern.
   *
   * CRITICAL: Both operations must start within 30ms of this call.
   * Use Promise.all — never sequential awaits.
   */
  async onUtteranceEnd(
    transcript: string,
    emotionHint: EmotionHint,
  ): Promise<void>;

  /**
   * Fired by Deepgram on partial transcript updates.
   * Feed into SentencePredictor to pre-warm LLM before utterance ends.
   */
  async onPartialTranscript(partial: string): Promise<void>;

  /**
   * Sends audio buffer to Twilio WebSocket.
   * Handles mulaw encoding and packet sizing automatically.
   * Thread-safe — can be called from filler and TTS simultaneously.
   */
  async sendAudioToTwilio(audio: Buffer, source: AudioSource): Promise<void>;

  /**
   * Graceful pipeline teardown.
   * Flush buffers, close Deepgram stream, stop TTS, log final state.
   * Never throws — always resolves.
   */
  async destroy(): Promise<void>;
}
```

### BackchannelEngine

```typescript
/**
 * Injects small acknowledgment sounds WHILE the customer is speaking.
 * This is the single most powerful "I'm listening" signal.
 *
 * Timing rules:
 * - Play every 8-15 seconds during continuous customer speech
 * - NEVER play two backchannels consecutively
 * - Volume: 15-20% lower than main AI voice
 * - Must play DURING their sentence, never in a gap (gaps = interruption)
 * - If customer pauses mid-sentence (< 1.5s), backchannel = invitation to continue
 */
class BackchannelEngine {
  private lastBackchannelAt: number = 0;
  private lastBackchannelId: string | null = null;
  private customerSpeakingStartedAt: number | null = null;

  /**
   * Called when customer speech is detected (VAD fires).
   * Starts the backchannel timing clock.
   */
  onCustomerSpeechStart(): void;

  /**
   * Called on every Deepgram partial transcript during customer speech.
   * Decides whether to inject a backchannel based on:
   * - Time since last backchannel (min 8s gap)
   * - Customer sentence continuity (don't inject at sentence boundaries)
   * - Current emotional context
   */
  async onPartialTranscript(
    partial: string,
    emotionState: EmotionState,
    pipeline: AudioPipeline,
  ): Promise<void>;

  /**
   * Called when customer finishes speaking.
   * Resets state for next turn.
   */
  onCustomerSpeechEnd(): void;

  /**
   * Inject backchannel at reduced volume.
   * Uses audio mixing — plays simultaneously with customer audio, not replacing it.
   */
  private async injectBackchannel(
    fillerId: string,
    pipeline: AudioPipeline,
  ): Promise<void>;
}
```

### SentencePredictor (LLM Pre-warming)

```typescript
/**
 * Predicts when a customer's utterance is near-complete and
 * pre-warms the LLM call before Deepgram fires utterance_end.
 *
 * This gives the LLM a 200-400ms head start, cutting perceived latency
 * significantly.
 *
 * Strategy:
 * - Track partial transcript growth
 * - Detect sentence-ending patterns (falling intonation, complete thought)
 * - Start LLM call with "likely intent" when 80% confident utterance is ending
 * - If prediction wrong, cancel and restart with correct transcript
 */
class SentencePredictor {
  /**
   * Update with latest partial transcript from Deepgram.
   * Returns prediction confidence and detected intent if high enough.
   */
  async update(partial: string): Promise<PredictionResult | null>;

  /**
   * Returns true if the partial transcript looks like a near-complete thought.
   *
   * Signals:
   * - Ends with question word + noun (likely question is done)
   * - Subject + verb + object pattern complete
   * - Common closing phrases: "you know?", "right?", "...like that"
   * - Partial has been stable for 300ms (no new words)
   */
  private isNearComplete(partial: string): boolean;

  /**
   * Pre-warm: start LLM call with current best-guess transcript.
   * Store the in-flight request so it can be cancelled if transcript changes.
   */
  async prewarmLLM(likelyTranscript: string): Promise<PrewarmedRequest>;

  /**
   * Called when actual utterance_end fires.
   * If prewarmed transcript matches final → use existing LLM stream (no latency).
   * If different → cancel prewarmed, start fresh (still faster than no prewarming).
   */
  async resolveWithFinal(finalTranscript: string): Promise<LLMStream>;
}
```

### CrossfadeManager

```typescript
/**
 * Manages the audio transition from filler → TTS response.
 * The crossfade must be seamless — customer should not hear a click,
 * pop, or silence during the transition.
 *
 * Crossfade algorithm:
 * - Detect natural pause point in filler audio (if possible)
 * - Begin 150ms linear volume fade-out of filler
 * - Begin 150ms linear volume fade-in of TTS
 * - Overlap the two by 100ms for smoothness
 * - Total transition: ~200ms, imperceptible
 */
class CrossfadeManager {
  /**
   * Execute crossfade from currently playing filler to new TTS stream.
   *
   * @param fillerPlayback - the in-progress filler audio
   * @param ttsStream - TTS audio stream (first chunk must already be buffered)
   * @param pipeline - to send mixed audio to Twilio
   */
  async execute(
    fillerPlayback: FillerPlayback,
    ttsStream: AsyncGenerator<Buffer>,
    pipeline: AudioPipeline,
  ): Promise<void>;

  /**
   * Find the nearest natural pause point in remaining filler audio.
   * Used to start crossfade at a musically natural moment.
   * Returns offset in ms from current position.
   */
  private findNaturalPausePoint(remainingAudio: Buffer): number;

  /**
   * Apply linear volume envelope to audio buffer.
   * direction: 'in' = silence → full, 'out' = full → silence
   */
  private applyVolumeEnvelope(
    audio: Buffer,
    direction: "in" | "out",
    durationMs: number,
  ): Buffer;
}
```

### DeepgramStreamingClient

```typescript
/**
 * Wrapper around Deepgram's real-time streaming API.
 *
 * Config for sales calls:
 * - Model: nova-2 (best accuracy + speed balance)
 * - Language: auto-detect initially, lock after first confident detection
 * - Endpointing: 300ms (how long silence before utterance_end)
 * - Smart formatting: false (we handle formatting ourselves)
 * - Interim results: true (needed for backchannel + prewarming)
 * - VAD events: true (needed for backchannel timing)
 *
 * IMPORTANT: Single WebSocket per call — never reconnect mid-call.
 * Buffer audio during any reconnection attempts.
 */
class DeepgramStreamingClient {
  /**
   * Initialize and connect to Deepgram.
   * Pre-warm the WebSocket connection before call connects to reduce
   * the first-utterance latency.
   */
  async connect(language: string): Promise<void>;

  /**
   * Feed raw mulaw audio from Twilio into Deepgram.
   * Called on every Twilio WebSocket message — must be non-blocking.
   */
  feed(audio: Buffer): void;

  /**
   * Emitted events (use EventEmitter pattern):
   *
   * 'partial'        → (transcript: string) — interim result
   * 'utterance_end'  → (transcript: string, confidence: number) — final
   * 'speech_start'   → () — VAD detected speech starting
   * 'speech_end'     → () — VAD detected speech ending
   * 'language'       → (lang: string, confidence: number) — language detected
   * 'error'          → (err: Error) — connection or processing error
   */

  /**
   * Detect language switch mid-call.
   * If customer switches language and Deepgram's new language confidence > 0.85,
   * emit 'language' event and switch STT model.
   * Target: < 100ms from first word in new language to detection.
   */
  async handleLanguageSwitch(newLanguage: string): Promise<void>;

  /**
   * Graceful close — flush any remaining audio before closing.
   */
  async close(): Promise<void>;
}
```

### ElevenLabsStreamingClient

```typescript
/**
 * Wrapper around ElevenLabs streaming TTS API.
 *
 * Config for maximum human quality:
 * - Model: eleven_turbo_v2_5 (lowest latency, multilingual)
 * - stability: 0.4 (lower = more expressive variation, sounds less robotic)
 * - similarity_boost: 0.8
 * - style: 0.3 (adds natural style variation)
 * - use_speaker_boost: true
 * - output_format: ulaw_8000 (matches Twilio format — no re-encoding needed)
 *
 * Streaming pattern:
 * - Send text sentence by sentence as LLM generates
 * - DO NOT wait for full LLM response
 * - First audio chunk target: < 250ms from first text sent
 */
class ElevenLabsStreamingClient {
  /**
   * Stream TTS for a text generator.
   * Yields audio Buffer chunks as they arrive from ElevenLabs.
   * Each chunk is immediately playable — no buffering needed.
   *
   * CRITICAL: Do not accumulate full text before sending.
   * Send each sentence to TTS as soon as LLM emits it.
   * This pipelines TTS with LLM generation — huge latency win.
   */
  async *streamFromGenerator(
    textGenerator: AsyncGenerator<string>,
    voiceId: string,
    toneConfig: ToneConfig,
  ): AsyncGenerator<Buffer>;

  /**
   * Apply tone configuration to ElevenLabs API params.
   * Maps our internal ToneConfig to ElevenLabs stability/style params.
   * Called before each TTS request based on current emotional state.
   */
  private applyToneConfig(config: ToneConfig): ElevenLabsParams;

  /**
   * Get voice ID for a given language.
   * Each language has a pre-cloned voice that sounds native.
   * Falls back to multilingual model if no dedicated voice exists.
   */
  getVoiceForLanguage(language: string): string;

  /**
   * Pre-warm the ElevenLabs WebSocket connection.
   * Keep-alive connection pool — one per language in use.
   * Eliminates TLS handshake latency on first TTS request.
   */
  async prewarmConnection(language: string): Promise<void>;
}
```

## CRITICAL CODING PATTERNS

### Pattern 1: Always Stream, Never Buffer Full Response

```typescript
// ✅ CORRECT — pipe LLM tokens directly to TTS
async function streamResponseToTTS(
  llmStream: AsyncGenerator<string>,
  ttsClient: ElevenLabsStreamingClient,
  pipeline: AudioPipeline,
) {
  // Sentence accumulator — send complete sentences to TTS
  let sentenceBuffer = "";

  for await (const token of llmStream) {
    sentenceBuffer += token;

    // When we have a complete sentence, send to TTS immediately
    if (endsWithSentenceBoundary(sentenceBuffer)) {
      const audioStream = ttsClient.streamFromText(sentenceBuffer);
      for await (const audioChunk of audioStream) {
        pipeline.sendAudioToTwilio(audioChunk, "tts");
      }
      sentenceBuffer = "";
    }
  }

  // Flush remaining text
  if (sentenceBuffer.trim()) {
    const audioStream = ttsClient.streamFromText(sentenceBuffer);
    for await (const audioChunk of audioStream) {
      pipeline.sendAudioToTwilio(audioChunk, "tts");
    }
  }
}

// ❌ WRONG — waiting for full LLM response before TTS
async function badPattern(llmClient, ttsClient, pipeline) {
  const fullResponse = await llmClient.getFullResponse(transcript); // blocks
  const audio = await ttsClient.synthesize(fullResponse); // blocks again
  pipeline.sendAudioToTwilio(audio, "tts");
}
```

### Pattern 2: Fail Soft — Never Crash a Call

```typescript
// ✅ CORRECT — graceful degradation
async function onUtteranceEnd(transcript: string) {
  try {
    await Promise.all([
      fillerManager.playImmediate("thinking").catch((err) => {
        logger.warn("Filler failed, continuing without it", { err });
        // Call continues — just without filler
      }),
      processWithLLM(transcript),
    ]);
  } catch (err) {
    logger.error("Pipeline error", { err, transcript });
    // Play a safe fallback response rather than silence
    await playFallbackResponse();
  }
}

// ❌ WRONG — letting errors crash the call
async function badOnUtteranceEnd(transcript: string) {
  await fillerManager.playImmediate("thinking"); // if this throws, call dies
  await processWithLLM(transcript);
}
```

```typescript
// ✅ CORRECT — emit and return immediately
onTwilioMessage(data: Buffer) {
```

### Pattern 4: Sentence Boundary Detection

```typescript
// Detect natural places to cut text for TTS chunking
function endsWithSentenceBoundary(text: string): boolean {
  // Send to TTS at sentence endings for natural delivery
  const patterns = [
    /[.!?]\s*$/, // Standard sentence endings
    /[,;]\s*$/, // Natural pause points (comma, semicolon)
    /\.\.\.\s*$/, // Trailing off (sends to TTS, sounds natural)
    / — \s*$/, // Em dash mid-sentence pause
  ];
  return (
    patterns.some((p) => p.test(text.trimEnd())) && text.trim().length > 15
  ); // Don't send tiny fragments
}
```

## STATE MACHINE

The pipeline tracks a strict state to prevent audio overlaps and race conditions:

```typescript
type PipelineState =
  | "idle" // Waiting for customer to speak
  | "listening" // Customer is speaking (VAD active)
  | "processing" // Utterance ended, filler playing + LLM thinking
  | "responding" // TTS audio playing to customer
  | "backchannel" // Injecting backchannel during customer speech
  | "crossfading" // Transitioning filler → TTS
  | "escalating" // Warm handoff in progress
  | "ended"; // Call has ended

// Valid state transitions only:
const VALID_TRANSITIONS: Record<PipelineState, PipelineState[]> = {
  idle: ["listening"],
  listening: ["idle", "processing", "backchannel"],
  backchannel: ["listening"],
  processing: ["crossfading", "responding"],
  crossfading: ["responding"],
  responding: ["idle", "listening"],
  escalating: ["ended"],
  ended: [],
};
```

## ERROR HANDLING REQUIREMENTS

Every external API call must have:

1. Timeout (Deepgram: 5s connect, ElevenLabs: 3s first chunk, Claude: 10s first token)
2. Retry with exponential backoff (max 2 retries on pipeline — call can't wait long)
3. Fallback behavior that keeps the call alive
4. Structured error logging with callId, timestamp, component, duration

```typescript
// Fallback response pool — pre-recorded for zero-latency playback
// Used when LLM or TTS fails mid-call
const FALLBACK_RESPONSES = [
  "Sorry, I lost my train of thought for a second — could you repeat that?",
  "Apologies, just a small hiccup on my end. What were you saying?",
  "Bear with me one moment... could you say that again?",
];
```

## WHAT COPILOT SHOULD NEVER DO IN THIS MODULE

- Never use `setTimeout` for audio timing — use audio buffer length calculations
- Never decode/re-encode audio unnecessarily — keep mulaw throughout
- Never block the Twilio WebSocket message handler with async work
- Never let a component error propagate to crash the call — always catch + degrade
- Never buffer the full LLM response before starting TTS
- Never play filler and TTS simultaneously without crossfade (audio clash)
- Never use polling to check if filler is done — use audio duration + callbacks
- Never hardcode voice IDs — always look up from language config map
- Never log raw audio buffers — log metadata only (duration, size, source)
- Never process audio synchronously on the main thread — use worker threads for heavy DSP
